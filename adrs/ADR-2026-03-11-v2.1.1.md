**Architectural Decision Record (ADR-2026-03-11-v2.1.1)**  
**Title:** Interactive User Interface – STDIN/STDOUT REPL Bridge + Sandboxed `user-agent` Skill  
**Status:** Approved (maniacal security + trivial auditing preserved)  
**Rationale:** The raw STDIN proposal is user-friendly but would bloat the microscopic TCB (`seedclaw.go`) with LLM orchestration, tool-calling, skill discovery, and response formatting. This violates Principles #1 (minimal committed core) and #2 (sandbox-first).  

**Selected Approach (most secure + user-friendly):**  
- `seedclaw` binary gains **only a 30-line thin bridge** (goroutine reading STDIN line-by-line).  
- Every user line is wrapped as a structured JSON message and forwarded **over the existing bidirectional TCP control channel** (already open from message-hub).  
- A new **sandboxed core skill** `user-agent` (generated during bootstrap, lives in Docker on `seedclaw-net`) owns the full agent loop:  
  - Maintains live skill registry + descriptions (queried from seedclaw via TCP).  
  - Calls `llm-caller` with system prompt containing tools=skills.  
  - Performs ReAct/tool-calling loop.  
  - Routes skill calls via message-hub.  
  - Formats final answer back to CLI.  
- All actions (user input, LLM calls, skill invocations, outputs) are logged immutably by seedclaw binary with full `network_policy` and `previous_hash`.  
- Zero new ports, zero new binaries, zero host-network exposure, zero audit mounts on any container.  

**Why this beats alternatives**  
- Full logic in binary → TCB explosion, impossible to sandbox. Rejected.  
- Separate `seedclaw chat` binary → dual-binary complexity, harder auditing. Rejected.  
- WebSocket/Telegram (old README) → external dependencies, non-terminal. Rejected.  
- Pure TCP client (user must run curl/json) → terrible UX. Rejected.  
- Result: `./seedclaw` (no flags) starts daemon + REPL in one terminal. User types natural language. Everything audited with one `grep`.  

**Security invariants preserved (non-negotiable):**  
- TCB remains `seedclaw.go` + four (now five) committed core skills.  
- `user-agent` runs with exact Default Container Runtime Profile + `"outbound": "none"`.  
- No skill ever sees raw STDIN; all input is JSON-validated.  
- Prompt injection attempts are contained inside Docker sandbox.  
- Trivial audit: every user request appears as `{"action":"user_request","skill":"user-agent",…}` in `shared/audit/seedclaw.log`.  

**Implementation sketch for `seedclaw.go` (will be enforced in bootstrap prompt):**  
```go
// Thin STDIN bridge – added in main()
go func() {
    scanner := bufio.NewScanner(os.Stdin)
    for scanner.Scan() {
        line := scanner.Text()
        if line == "" || line == "exit" { continue }
        msg := ControlMessage{
            From: "user",
            To:   "user-agent",
            Content: map[string]string{"action": "user_request", "prompt": line},
        }
        sendOverTCP(msg) // reuses existing bidirectional conn to message-hub
    }
}()
```
